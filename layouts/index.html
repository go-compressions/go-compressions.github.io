Why go-compressions

Despite the name, this org houses the broader family of byte-stream transformation primitives — both lossless compression codecs and content-addressable hash functions. They live together because they sit at the same layer of the stack : pure-Go, allocation-aware, zero-copy-friendly primitives that everything above them builds on. Different math, same place in the stack. A storage layer hashes its blobs and compresses its bodies side by side ; it's natural to ship those primitives side by side too.

Two concerns, one layer

Compression codecs

Pure-Go implementations of stream compressors. LZFSE / LZVN today — Apple's algorithm, where LZFSE is LZVN followed by an entropy stage. More codecs land as concrete callers need them.

Content-addressable hashes

Pure-Go implementations of modern cryptographic / keyed hashes. BLAKE3 today — hash, keyed hash, KDF, all spec-conformant against the reference test vectors.

Why together

Same allocation discipline, same multi-arch story, same fuzz / cross-check methodology. A content-addressable storage stack reaches for both; shipping them in one org keeps the engineering bar consistent.

Repositories

Compression codecs

lz4

Pure-Go LZ4 block codec, wire-compatible with pierrec/lz4. Honest framing: it matches the reference on ratio and trails the fastest Go libs on raw speed; its match-finder delegates to go-simd/matchlen, whose SIMD common-prefix kernel runs on all six of Go's 64-bit targets.

lzfse

Pure-Go LZFSE / LZVN codec. LZFSE = LZVN + entropy stage, mirroring Apple's reference C implementation.

lzfsec

CLI wrapper around lzfse, mirroring the interface of Apple's reference lzfse binary.

Content-addressable hashes

blake3

Pure-Go BLAKE3 hash, keyed hash, and KDF. Cross-checked against the reference test vectors.

b3sum

Pure-Go b3sum CLI matching the reference Rust b3sum.

Docs

docs

Org-level documentation, design notes, NEON / archsimd notes.

Quality bar

  • Pure Go. CGO_ENABLED=0 across the org. No cgo bindings, no native shims.
  • 100% test coverage. Every module is gated on it. Codecs are fuzz-tested against upstream reference implementations ; hashes are cross-checked against spec test vectors.
  • Allocation-aware. Hot paths avoid per-call allocations ; APIs expose buffered variants where it matters.
  • Multi-arch. Pure-Go reference implementations ; SIMD / NEON acceleration layered behind build tags where it makes sense.
  • BSD-3-Clause on every source file.

Who uses it

cloud-boot uses lzfse as the body compressor for its PE32+ / UKI packer (go-coff/efipack). The broader content-addressable storage stack uses blake3 as the address function for layer / blob lookup. Both consumers drove the original split — the codecs and the hashes already had to ship together.
